Creating an HTML Add-in

HTML add-ins are implemented as HTML, ASP, or ASPX pages that can include VBScript or JScript code.

To create an HTML add-in:

  1. Install Microsoft Retail Management System Store Operations.

  2. Create an HTML file that uses VBScript or JScript code to access the Store Operations business logic through the QSBridge ActiveX control. For more information about QSBridge, see The QSBridge ActiveX Control in this topic.

  3. Save your HTML script to the HTML folder specified under the Paths tab of the Administrator Configuration. The default location is C:\Program Files\Microsoft Retail Management System\Store Operations\HTML.

  4. Test and debug your script.

 

The QSBridge ActiveX Control

The QSBridge ActiveX control connects HTML script add-ins to the Store Operations. This control provides access to the business rules contained in QSRules and invokes macro-level events in the POS application to trigger specific functions. When the POS application detects a change in the HTML content, for example, redirection, it searches the HTML content for an instance of the QSBridge ActiveX control. If an instance is found, it creates a bridge receipt and sets a reference (with events) to that instance. The bridge receptor allows the POS application to receive and respond to any events that are fired by the QSBridge control running in any of the HTML browsers.

The POS bridge receptor responds to the following functions called through QSBridge.

  • QSBridge.FireEvent(String, ParamArray): Invokes the event specified by the String parameter, with the optional parameters specified in the ParamArray parameter. This function resides in the user interface workflow layer. For more information, see Using the FireEvent Function.

  • QSBridge.RequestQSRules(): Returns a reference to the current SessionClass object within QSRules. For more information, see Using the RequestQSRules Function.

Remember the following points when working with the QSBridge control:

  • QSBridge can only be used by HTML script add-ins.

  • QSBridge events trigger logic that occurs in the user interface workflow. We suggest using the FireEvent function when possible because events operate at a higher level than the QSRules classes.

  • QSRules events often activate the same user interface workflow that the POS uses. For example, the workflow logic triggered by the PerformAddItem event is the same code that is called when an item is scanned in the POS.

To see how the loyalty add-in uses the QSBridge ActiveX control, see Loyalty Information Display Sample.
